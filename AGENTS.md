## Workflow (OpenSpec-first, SSOT)
`openspec/` is SSOT.
1. Find relevant `openspec/specs/**`, treat as SSOT.
2. Behavior/requirements/contracts/schema change → create `openspec/changes/**` FIRST (proposal → tasks).
3. Implement; keep code + `spec.md` in sync.
4. Validate: `openspec validate --specs`.
5. Verify + archive when done; never archive unverified.

Layout: active `openspec/changes/<change>/`; main `openspec/specs/<capability>/spec.md`; archived `openspec/changes/archive/YYYY-MM-DD-<change>/`.
Commands: `/opsx:new|continue|ff|apply|verify|sync|archive`.

## Documentation
- Do NOT add/update feature/behavior docs under `docs/`; use OpenSpec context docs (`openspec/specs/<capability>/context.md` or change-level `context.md`/`notes.md`) as SSOT.
- Do NOT edit `CHANGELOG.md` directly (release process owns it).
- `spec.md` = normative testable requirements ONLY; `context.md` = purpose/rationale/examples/ops (split into overview/rationale/examples/ops if it grows); sync stable change context back to main context docs.

## Contributing & Merge Gates
Binding workflow: `.github/CONTRIBUTING.md`.
- Merge gates: CI green + `@codex review` clean (or findings addressed) + `mergeable=CLEAN` + OpenSpec folder for behavior changes + `Fixes #N`/`Closes #N`.
- No self-merge by default; split large PRs (~1 concern, ~800 net lines). Self-merge escape hatch only after 14 days with all gates met + a comment invoking the clause.
- Assistant preparing a merge MUST verify gates against live GitHub state (status rollup, codex review submissions, `mergeable`), NOT local history. Local `pytest`/`ruff`/`codex review` encouraged but not substitutes.

## PR Readiness / Review Trapdoors
- OpenSpec is a HARD gate for behavior/API/schema/CLI/dashboard/proxy-routing/operator-contract/compat changes. Create `openspec/changes/<slug>/` before coding; `spec.md` normative with MUST/SHALL; rationale/examples in `context.md`; run strict validation before "ready". Code/tests alone insufficient.
- Codex review state must come from current-head GitHub evidence (labels, latest review/comment/reaction, GraphQL threads). Usage-limit/env/missing-review = missing evidence, NOT approval. Unresolved non-outdated P-level threads block readiness even if the top-level comment looks clean.
- Proxy failover/retry: prove account ownership + settlement invariants. File-pinned requests must NOT cross accounts; API-key reservations settle before error-health writes; excluded accounts must leave the selection loop; idle disconnects must NOT mark healthy accounts unhealthy; trusted-access routing degrades only along the documented path.
- Async/fan-out/session-lifecycle: prove task ownership + cleanup. Do NOT share one `AsyncSession` across concurrent tasks; cancel/await spawned tasks on failure; preserve finalization/settlement after partial errors; bound fan-out; test partial-failure, not only all-success.
- Migrations: single-head Alembic graph on the correct parent; up/down coverage where expected; backfill/compat for historical rows when new fields affect existing data.
- Issue-resolving PRs name exact `Fixes #N`/`Closes #N` (or state partial). One concern wide. Revive stale work as a focused branch on current `main`; don't drag old broad/conflicted branches forward unless the maintainer wants that.
- Bug fixes need regression coverage at the externally failing path (route/bridge/websocket/CLI/schema/dashboard/migration); helper-only tests insufficient when the failing surface is elsewhere.
- Compat work: verify canonical AND equivalent paths, trailing-slash behavior, external error envelopes, env-var semantics, response-schema contracts; update OpenSpec/context + tests together.
